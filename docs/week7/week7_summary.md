목표
6주차에 구축한 VPC/EC2 기반 인프라 위에 관리형 데이터베이스(RDS), 객체 스토리지(S3), 로드밸런서(ALB)를 추가하여 맘편한 이동 인프라를 완성하는 것을 목표로 한다. 지도교수 피드백인 HTTPS 통신 보안을 반영하여 도메인 구매, ACM 인증서 발급, HTTPS 리스너 구성까지 진행한다. 보안 그룹 체인(ALB→EC2→RDS)을 설계하여 각 계층의 접근을 최소화하고, Elastic IP를 할당하여 EC2의 고정 IP를 확보한다. RDS를 프라이빗 서브넷에 배치하여 외부 접근을 차단하고, S3 퍼블릭 액세스를 완전 차단하여 데이터 보안을 확보한다.

진행내용
RDS PostgreSQL 15 인스턴스를 프라이빗 서브넷에 생성하고, DB 서브넷 그룹과 커스텀 파라미터 그룹(SQL 로깅)을 구성하였다. S3 버킷에 버전관리, AES-256 암호화, 퍼블릭 액세스 완전 차단, CORS를 설정하고, EC2 IAM 역할로 Access Key 없이 접근하도록 구성하였다. ALB를 생성하고 타겟 그룹에 EC2를 등록하여 헬스체크(/api/v1/health)를 설정하였다. 보안 그룹 3개를 생성하여 ALB(80,443)→EC2(8000)→RDS(5432) 체인을 구성하였다. comfortablemove.com 도메인을 구매하고 Route53 호스팅 영역을 생성하였다.

진행결과
ALB를 통해 HTTPS(443) 접속이 가능하고 HTTP(80)는 HTTPS로 자동 리다이렉트됨을 확인하였다. ACM에서 SSL 인증서를 무료 발급하고 DNS 검증을 완료하여 ALB HTTPS 리스너에 연결하였다. RDS는 프라이빗 서브넷에서 EC2에서만 접근 가능하고 스토리지 암호화가 적용됨을 확인하였다. S3 버킷은 퍼블릭 액세스가 완전 차단되고 서버 측 암호화가 적용되었다. Elastic IP(3.37.165.32)를 할당하여 EC2 고정 IP를 확보하였다. 전체 33개 AWS 리소스를 구성하여 프로덕션 수준의 인프라를 완성하였다.

기타
가장 큰 성과는 보안 그룹 체인과 HTTPS를 포함한 전체 보안 아키텍처를 완성한 점이다. ALB→EC2→RDS 체인으로 각 계층의 접근을 SG ID 기반으로 제한하여 IP 변경에도 안정적이다. HTTPS 구성에서 도메인, Route53, ACM, ALB 리스너가 모두 연결되어야 한다는 전체 흐름을 이해하였다. CloudWatch 대시보드에 EC2 CPU, ALB 요청, RDS 연결 등 6개 위젯을 구성하고, CPU 과부하, 5xx 에러, 헬스체크 실패, 스토리지 부족 등 5개 알람을 설정하여 운영 가시성을 확보하였다. ALB가 프리티어에 미포함되어 월 약 $16 과금되는 점을 확인하고 테스트 후 정리 계획을 수립하였다.
