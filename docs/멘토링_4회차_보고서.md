### [활동 내용]

#### 1) GitHub Actions CI 실증 검증 및 Jenkins CI 구축 (13~14주차)
* 2·3회차에서 "CI 파이프라인이 실제로 동작하는지 정량적으로 보여주지 못한다"는 점을 보완하기 위해, GitHub Actions 워크플로가 push·Pull Request 트리거에서 실제로 기동되는지 검증함. flake8 정적 분석과 pytest 테스트가 매 실행마다 완주하고, pip 의존성 캐시가 적용된 실행에서 설치 시간이 단축되는 효과를 정량 데이터로 측정하였으며, 빌드된 이미지가 ECR에 SHA·latest 두 태그로 저장되는 것을 `aws ecr describe-images`로 확인함.
* GitHub Actions와의 비교를 위해 Jenkins 기반 CI를 별도로 구축함. JCasC(Configuration as Code)로 설정을 코드화하고, multibranch 파이프라인과 SCM 폴링으로 브랜치별 자동 빌드를 구성하였으며, 동일한 flake8·pytest·이미지 빌드 단계를 Jenkins에서도 통과시켜 두 CI 도구의 구성 방식과 장단점을 직접 비교함.

#### 2) AWS 클라우드 프로덕션 배포 및 iOS 앱 연동 (14주차)
* 3회차에서 핵심 과제로 남았던 "minikube 로컬 환경에 머물러 상시 가동되는 서비스를 보여주지 못한다"는 문제를 해결함. 기존 Terraform 모듈(네트워크·보안·컴퓨트·RDS·인그레스·DNS·모니터링)을 그대로 적용하여 AWS에 풀스택을 프로비저닝하고, 보유한 `comfortablemove.com` 도메인(Route53)과 ACM 인증서를 연결하여 HTTPS로 상시 접근 가능한 상태를 구성함. 운영 환경의 스키마 관리를 위해 Alembic 마이그레이션을 도입하여 컨테이너 시작 시 자동 적용되도록 함.
* iOS 앱이 그동안 로컬 환경에만 연결되던 것을, 클라우드 백엔드(`https://comfortablemove.com`)를 바라보도록 설정을 분리(BackendConfig)하고, BLE 기기명을 한글 노선명에서 영문 식별자로 변환하는 매핑(DistrictMapper)을 정비하여 실제 탑승 기록이 클라우드 DB에 저장되도록 연동을 완료함. 또한 Ansible로 서버 공통 설정·Docker 환경 프로비저닝을 코드화하여 재현 가능한 서버 구성을 확보함.

#### 3) EKS·ArgoCD GitOps 상시 운영 체계 구축 (15주차)
* 단일 서버 배포를 넘어, EKS 관리형 쿠버네티스 클러스터(워커 노드 2대, K8s v1.33)를 프로비저닝하고 백엔드·PostgreSQL·Redis를 파드로 배포하여 상시 가동 상태를 구성함. 헬스 체크 경로를 외부 API를 호출하지 않는 `/health/ready`로 분리하여, 11주차에 발견했던 외부 API 일일 한도 소진 누수가 클러스터 환경에서도 재발하지 않도록 함.
* ArgoCD를 도입하여 Git 저장소의 매니페스트를 단일 진실 공급원(Single Source of Truth)으로 삼는 GitOps 방식의 지속적 배포(CD)를 구성함. 자동 동기화(automated sync)·불필요 리소스 정리(prune)·수동 변경 자동 복구(selfHeal)를 활성화하고, 매니페스트의 replica 수를 변경하면 수십 초 내 클러스터에 자동 반영되는 것, 이전 리비전으로 롤백되는 것, 배포 이력이 보존되는 것을 직접 확인함. 현재 ArgoCD 대시보드에서 애플리케이션이 Synced/Healthy 상태로 표시됨.

### [멘토링 논의 사항]

#### 1) 실제 버스 탑승 환경 시연 영상의 필요성
* 그동안의 검증이 정류장·실내 등 통제된 환경에서 이루어져, 서비스가 실제 버스 안에서 의도대로 동작한다는 실증이 부족하다는 지적을 받음. 지도교수께서 실제 버스 탑승 환경에서 BLE 비콘 감지 → 배려석 알림 전송 → 클라우드 DB 저장까지의 연속 흐름을 영상으로 확보해야 심사자와 사용자 모두에게 서비스의 신뢰를 줄 수 있다고 조언함.

#### 2) 앱스토어 배포 버전 정합성 확인
* 백엔드·인프라가 로컬에서 클라우드(프로덕션 도메인)로 대폭 전환된 만큼, 현재 앱스토어에 배포된 iOS 앱 버전이 변경된 클라우드 백엔드와 정합하게 동작하는지 점검이 필요하다는 의견을 받음. 앱스토어 등록 정보(업데이트 내역)에도 이번 변경 사항이 반영되어야 함을 확인함.

#### 3) 클라우드 운영 비용 최적화는 후속 과제로 분리
* EKS 클러스터·노드·RDS 등 상시 가동에 따른 비용이 발생하나, 현 단계는 기능 완성과 실증이 우선이라는 데 의견이 일치함. 비용 최적화(노드 수 축소, 스팟 인스턴스, 오토스케일링, 비가동 시 축소 등)는 프로젝트 종료 후 애프터(after) 단계의 별도 개선 과제로 다루기로 함.

#### 4) 개인정보 보호 및 유출 가능성 점검
* 서비스가 수집·저장하는 데이터에 개인정보 유출 위험이 없는지 점검하고, 식별된 위험에 대해 실제 보완 조치를 적용함. 점검 결과 이름·연락처·계정 등 직접적인 개인 식별 정보(PII)는 수집하지 않으며, 기기 식별은 무작위 생성 UUID(`device_id`) 기반의 가명정보 성격임을 확인함. 다만 두 가지 위험을 추가로 식별함. 첫째, 출산 예정일·임산부 인증 여부는 임신 주차별 맞춤 알림을 위한 건강 관련 선택적 민감정보에 해당함. 둘째, 탑승 위치(위·경도)와 시각이 동일 `device_id` 단위로 축적될 경우 이동 패턴이 형성될 수 있음. 이를 보완하기 위해 (1) 탑승 위치를 정밀 좌표 대신 약 100m 단위(소수점 3자리)로 절삭하여 최소 수집하도록 백엔드 코드를 보강하고, (2) 수집 항목·목적·보관·파기·보호조치·제3자 미제공을 명시한 개인정보 처리방침을 수립함. 아울러 모든 통신 구간은 HTTPS(TLS)로 암호화되고, 데이터베이스는 프라이빗 서브넷에 격리되어 외부에 직접 노출되지 않으며 저장 시 암호화됨. 향후 보관 기한 경과 시 자동 파기와 통계 집계 위주 활용으로 재식별 가능성을 최소화함.

## 학생 최종의견

문제점: 13~15주차에 걸쳐 CI 파이프라인을 실증 검증하고, 서비스를 minikube 로컬에서 AWS 클라우드 프로덕션(상시 가동 도메인)과 EKS·ArgoCD GitOps 운영 체계로 전환하였으며 iOS 앱을 클라우드 백엔드와 연동 완료함. 그러나 ① 실제 버스 탑승 환경에서의 End-to-End 동작을 담은 시연 영상이 아직 없어 서비스의 실증적 신뢰가 부족하고, ② 백엔드 클라우드 전환 이후 앱스토어 배포 버전과의 정합성이 확인되지 않았으며, ③ 수집 데이터의 개인정보 처리 방침이 문서로 정리되어 있지 않음.

대책방안: ① 실제 버스 노선에 탑승하여 BLE 비콘 감지 → 배려석 알림 → 클라우드 DB 저장이 연속으로 이어지는 과정을 영상으로 촬영하고, 저장 결과를 통계 API와 DB 조회로 즉시 확인하는 라이브 검증을 함께 보임. ② 앱스토어 등록 버전이 클라우드 백엔드와 정합하게 동작하는지 점검하고, 필요 시 업데이트 내역을 반영함. ③ 수집 항목을 재점검하여 출산 예정일 등 건강 관련 민감정보와 탑승 위치 축적 위험을 식별하고, 탑승 위치를 약 100m 단위로 절삭 수집하도록 백엔드를 보강하였으며, 개인 식별 정보 미수집·전 구간 HTTPS 암호화·위치 최소화·통계 집계 위주 활용을 골자로 하는 개인정보 처리방침을 수립함. 클라우드 운영 비용 최적화는 프로젝트 종료 후 별도 개선 과제로 분리함.

추진 예정 사항: 최종 발표 전까지 실제 버스 탑승 시연 영상을 확보하고, 앱스토어 배포 버전의 정합성 점검을 완료하며, 개인정보 보호 점검 결과를 문서화함. 프로젝트 종료 이후에는 EKS 노드 축소·스팟 인스턴스·오토스케일링 등 클라우드 비용 최적화와, 기기 메타데이터(기기명·OS 버전 등)의 정식 등록 흐름 보강을 애프터 개선 과제로 추진할 예정임.
